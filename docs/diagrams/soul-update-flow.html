1. 事件来源矩阵

来源 是否进入 Pipeline 触发场景 后续影响
XHS bootstrap_profile 进入 插件回传 saved / liked / xhs_history，且不是 init 正在运行。 转成 favorite / like / view 后进入 signals_from_events()。
Douyin bootstrap_profile 进入 插件回传 dy_post / dy_collect / dy_like / dy_follow。 转成 view / favorite / like / follow，再分到 Surface / Interest / Role。
YouTube bootstrap_profile 进入 插件回传 yt_history / yt_subscriptions / yt_likes。 转成 view / follow / like，写入 pipeline buffer。
B 站账户同步 旁路 / 定时 AccountSyncService 默认每 6 小时同步 history / favorites / following。 新增事件直接进入 analyze_events() 更新 preference；若 soul 为空，会自动 bootstrap 一次。
推荐点击 进入 /api/recommendation-click 记录一次用户点开推荐卡。 构造 RECOMMENDATION_CLICK 强信号，路由到 Interest + Surface。
普通 /api/events 不进入 插件批量上报普通行为事件。 只落 event list 并触发 refresh；当前没有接到 pipeline。
推荐反馈 /api/feedback 旁路 用户 like / dislike / comment / dismiss 推荐。 即时 cognition card；feedback batch 达阈值后 Preference Analysis，显著变化才 rebuild。

兴趣 / 避雷探针 旁路 + tick 用户确认、拒绝或通过 chat 回应主动探针。 先更新 speculator 状态与 cognition；确认数达标后由 force_tick / pipeline tick 转正写回兴趣或避雷。
手动画像编辑 读时覆盖 /api/profile/edit 或画像编辑 UI 修改兴趣、避雷、价值观等字段。 写 profile_overrides.json，effective profile 立即变化；新增 dislike 还会异步清池。
画像整理 定时 / 手动 pipeline tick 中的 ProfileConsolidator，或 profile-consolidate --apply。 合并重复 likes / dislikes，重建 Onion 兴趣树并追加 changelog / cognition。
Bili source task-result 不进入 Bili 扩展搜索任务回传结果。 主要写 discovery candidates，不作为画像 bootstrap 信号。
X fetch / YouTube Takeout 不进入 CLI 导入 X 点赞收藏或 YouTube Takeout。 写 memory，部分直接跑 analyze_events()；不走 pipeline。

2. 后台与定时触发器

ContinuousRefreshController.run_forever()

后端启动后挂起多条 asyncio loop，不使用传统 cron。与 soul 直接相关的是 _loop_soul_pipeline()。

60s
默认每 refresh_check_interval_seconds 调一次 pipeline.tick()。
gate
受 scheduler.enabled、插件断开暂停和 init-active gate 控制。
cron
discovery_cron 只是旧配置兼容字段，当前 runtime 不消费。

ProfileUpdatePipeline.tick()

tick 是“定时成立”的核心：没有新 ingest 时也会检查 buffer、探针生命周期和低频维护。

buf
达到每层 min_signals + min_interval 后局部写回 Surface / Interest / Role / Values / Core。
30m
空闲时默认每 30 分钟跑 speculator 生命周期；有 layer 更新时立即跑。
12h
CognitionCycle 同步 awareness / insight 窗口；ProfileConsolidator 整理重复兴趣/避雷。

AccountSyncService.run_forever()

账户同步是另一条后台循环，独立于 discovery refresh。

6h
默认 account_sync_interval_hours=6，增量拉 B 站历史、收藏、关注。
pref
有新增事件时写 memory，再调用 analyze_events() 更新 preference。
boot
如果 soul 还没初始化，每进程最多自动 build_initial_profile([]) 一次。

配置热重载后的 one-shot

保存配置并重启后台任务后，会补一次不阻塞响应的探针生成。

+
post_reload_speculate / post_reload_avoidance_speculate 调 force_tick()。
?
这通常生成或轮换 probe；只有已有确认满足条件时才会间接写回兴趣/避雷。

3. Pipeline 内部分层路由

signals_from_events()

普通 event 进入 pipeline 时先被压成两类信号：

1
like / coin / favorite / comment -> ENGAGEMENT_EVENT
2
其它 event，例如 view / follow / click -> BEHAVIOR_EVENT
3
事件 payload 原样保留，供 PreferenceAnalyzer 和 LLM prompt 读取。

强信号 recommendation_click

推荐点击不是从 event list 拉出来的；API 直接构造强信号。

1
先落一个普通 click event，供历史和 awareness 使用。
2
再构造 RECOMMENDATION_CLICK，进入 Interest + Surface。
3
强信号让对应 layer 的 min_signals=1，但最短间隔仍生效。

4. 各层如何处理事件

Surface

纯规则，无 LLM。当前只统计 view/search，用 EMA 调整 depth_preference。

min 3 5 min rules

Interest

调用 PreferenceAnalyzer，保存 flat preference，再 populate_from_flat_preference() 写回兴趣树。

min 3 10 min LLM

Role

LLM delta prompt 判断 life_stage/current_phase 是否变化，只有 changed=true 才写。

min 5 24 h LLM delta

Values

LLM delta prompt 更新价值观与驱动力。Values 变化后会触发 portrait 重生成。

min 5 24 h portrait trigger

Core

最强保护层，LLM delta prompt 更新核心特质、深层需求和 MBTI。Core 变化也触发 portrait。

min 8 48 h strong guard

Portrait

不是独立 buffer 层。Core 或 Values 变化时，只重新生成 personality_portrait 文本。

derived LLM build

5. 典型更新场景

场景 A：点开推荐卡

1
API 落一个普通 click event。
2
额外推入 RECOMMENDATION_CLICK 强信号。
3
Interest 调 PreferenceAnalyzer；如新增兴趣或权重变化，则写 Soul。
4
Surface 被触发，但当前单击通常不改 Surface。

场景 B：重新拉小红书收藏

1
插件回传 bootstrap_profile task-result。
2
后端去重、过滤自 authored notes、转成 favorite / like / view。
3
如果不在 init 中，调用 _ingest_profile_update_events()。
4
Pipeline 按层 buffer，ready 后局部更新 Soul。

场景 C：反馈 dislike

1
写 feedback event，并立即写 cognition card。
2
不进入 pipeline。
3
新 feedback 达批量阈值后跑 PreferenceAnalyzer。
4
preference 显著变化才完整 rebuild Soul。

场景 D：对话说“我在备考”

1
DialogueInsightAnalyzer 抽出 goal 候选。
2
候选写 insight_candidates.json，先不改 Soul。
3
达到 confidence 或 occurrences 阈值后进入 PreferenceAnalyzer。
4
preference 显著变化才完整 rebuild Soul。

场景 E：Awareness / Insight

1
pipeline tick 调 CognitionCycle.run_if_due()。
2
Awareness 从 event memory 读新事件。
3
Insight 从新 awareness 中生成假设。
4
只同步最近窗口到 Soul，不触发完整 rebuild。

场景 F：手动加避雷

1
写 profile_overrides.json，effective profile 立刻生效。
2
如果新增 dislike，后台调度 pool purge。
3
raw Soul 不一定改，但业务读取时会叠加覆盖层。

场景 G：后台账户同步

1
account sync loop 到期拉新增 history / favorites / following。
2
新增信号逐条写入 memory event list。
3
直接调用 analyze_events() 更新 preference。
4
若 soul 为空，自动补一次低配初始画像；已有画像不会每 6 小时强制 rebuild。

场景 H：12 小时画像整理

1
pipeline tick 进入 ProfileConsolidator.run_if_due()。
2
规则合并、embedding 聚类、LLM 裁决重复主题。
3
写回 preference / Onion 兴趣树，穿透更新 overrides。
4
记录 consolidation cognition 和 soul_changelog.md。

6. 当前实现边界

注意：普通 Event 表没有“累计到阈值就统一触发 Preference Analysis”的后台机制。当前通用事件表主要服务 memory、awareness 和推荐 refresh；真正会触发 Preference Analysis + 可能 rebuild 的后台阈值路径是 feedback batch、dialogue learning，以及账户同步直接调用的 analyze_events()。后端调度也不是 cron 驱动，而是 runtime 内部 asyncio loop。
