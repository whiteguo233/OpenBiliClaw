推荐模块不自己发现内容，只消费 Discovery 填好的候选池 content_cache。下面把它的主要流程拆成步骤讲清楚。英文术语首次出现时用 等宽蓝字 标出并就地加中文解释，完整对照见 §8。一句话总览：先在后台把候选补好分类、文案、惊喜分，要展示时再二次评分 + 多样性选取挑出一批，写进历史并标记「已展示」。

A. serve() 热路径 — 怎么从候选池挑出一批推荐（10 步）

三个入口（生成 generate / 换一批 reshuffle / 续页 append）都委托给同一个 serve()。

  1. 读库存：count_pool_readiness 给出 可用(available) / 原始(raw) / 待补(pending) 三态，避免把还没准备好的算成「可换」。
  2. 取候选窗口：get_pool_candidates，所有门槛都写在 SQL 里 —— 必须 fresh、非 dislike、有分类、有文案、有可跳转 URL、不在近期已看、排除小红书你自己发的。
  3. 排除当前页（仅续页）：传入 excluded_bvids，避免续页重复展示已经在屏幕上的内容。
  4. 排除近期已看：按最近看过的 bvid 再过滤一遍。
  5. 合并近义话题：_merge_topic_supergroups 查一张预热好的「话题超组」映射表，把「动漫 / 动漫产业 / 动漫文化」合成一个桶。纯查表，热路径不现算向量。
  6. 二次评分（可选）：PoolCurator 给每条算综合分覆盖原排序（见 C）。
  7. 取向量：_fetch_candidate_embeddings，只读缓存，绝不在这一步调向量接口；缓存没命中就退回字符串方案。
  8. 多样性选取：有向量走 MMR（在「相关度高」和「与已选不重复」之间权衡），没向量退回字符串限量贪心；两条路都受 话题 / 风格 / 广义话题 / 放大 上限约束。
  9. 写历史：batch_insert_recommendations 单事务插入（唯一的同步写，要拿到 ID 才能返回）。
  10. 文案 + 标记：生成模式逐条 LLM 写文案；换一批 / 续页直接用池内预生成文案。最后脱离式地把选中项标记为 shown（已展示），不占用响应时间。
B. 后台预计算 — 分类 / 文案 / 惊喜
  1. 源无关分类 classify_pool_backlog：给跨源（小红书 / 抖音扩展）那些没分类的内容补上 风格 / 话题 / 相关度，用的是和发现侧一样的评估提示（含「近期反例」）。每批 10 条。补完之后，推荐层就只看内容特征、对来源平台透明。
  2. 预生成文案 precompute_pool_copy：后台批量写「推荐语 / 话题标签」，默认每批 30 条、2 worker 并发；解析失败先拆半重试，遇到限流就留空、下一轮重试，不卡住。
  3. 惊喜打分 precompute_delight_scores：先免费粗筛相关度≥0.55 取前 50，再用 LLMDelightScorer 按「跨域惊喜 / 戳隐藏需求」的标准打分，过阈值（0.70；探索开放度低则 0.80）才留下理由和钩子。
  4. 三者各自独立加锁，互不阻塞 —— 文案队列短不会饿死惊喜打分。
C. 排序与多样性 — PoolCurator 五维 + MMR + 放大保护
  1. 五维综合分 rec_score = 相关度×0.30 + 新鲜度×0.20 − 话题疲劳×0.25 − 来源单调×0.15 + 惊喜度×0.20，再叠加反馈调整（踩 UP −0.20 / 踩话题 −0.10 / 喜欢话题 +0.05…），最后超预算方向 −0.35，分数下限 0。
  2. MMR 多样性：每一步都选「相关度高 且 和已选内容不太像」的，两者各占 0.5；相似度用向量余弦，能抓到字符串看不出的「同题不同标签」重复。
  3. 数量上限：每话题 1~2 条、每风格 max(1,min(3,…))、广义话题 2/3/4（最硬，几乎不放宽）。
  4. 放大保护：刚确认的新兴趣，若 24 小时内占比≥0.25 算「超预算」，每批最多 floor(limit×0.25) 条，防止一个新方向刷屏。⚠ 注意：当前 serve() 没有注入「新确认方向」，所以这道保护默认是休眠的，需由别处显式注入才生效。
D. 三个入口的区别
  1. generate_recommendations（生成，limit 10）：realtime 模式 —— 插库后逐条调大模型写文案，质量高但慢。
  2. reshuffle_recommendations（换一批，limit 5）：precomputed 模式 —— 直接用池里预生成的文案，秒出。
  3. append_recommendations（续页，limit 10）：precomputed + 传入「当前页 bvid」避免重复。
E. 反馈闭环 — 点击 / like / dislike / dismiss
  1. 点击 /api/recommendation-click：优先用回传的真实 URL + 平台（避免跨源 ID 被错拼成 B 站链接）；点击是「强信号」，进画像 + 进短期探索缓冲。
  2. like / comment：进偏好与短期缓冲；攒够阈值会触发偏好重析（→ Soul）。
  3. dislike：作为强负样本进「近期反例」+ 偏好避让；delight 的 dislike 会直接把该条标记为 purged_by_dislike（从池里清掉）。
  4. dismiss（忽略）：软移除 —— 只标 feedbacked，不更新画像、不升级成话题级负反馈。两条 dismiss 路径略有差别：普通反馈的 dismiss 会发一条反馈事件并标 feedbacked；delight 的 dismiss 只标「已通知」。
F. delight 主动惊喜推送
  1. 每 120 秒一轮：备好惊喜候选 → get_pending_delight（要求分≥0.70、距上次推送≥4 小时冷却、硬过滤掉你已讨厌的话题）。
  2. 推送：通过 WebSocket 给插件发一条 delight.candidate（单条，受冷却约束）和 delight.refreshed（净新增数）。
  3. delight = 主动「惊喜」推荐：不是常规相关推荐，而是跨域、可能戳中你隐藏需求的内容。

① 候选池与就绪
content_cache候选池：一张 SQLite 表，存放所有等待推荐的内容（由 Discovery 与各来源填充）。
pool_status池中每行的状态：fresh 新鲜 / shown 已展示 / feedbacked 已反馈 / purged_by_dislike 被避雷清除。
count_pool_readiness库存三态统计：available 可用 / raw 原始 / pending 待补。
get_pool_candidates取候选窗口，所有过滤门槛都写在 SQL 里。
readiness gates 就绪门槛可换候选必须：有分类 + 有文案 + 有可跳转 URL + 不在近期已看 + 非 dislike。
bvid / content_idB 站视频 ID；跨源内容用 content_id 标识。
② serve 热路径
RecommendationEngine.serve()推荐主入口，三个入口都委托给它。
generate / reshuffle / append生成 / 换一批 / 续页 三种调用。
expression_mode realtime / precomputed文案模式：实时逐条调大模型 / 直接用池内预生成。
excluded_bvids续页时传入「当前页已展示的 ID」，避免重复。
_merge_topic_supergroups合并近义话题（查预热好的映射表，热路径不现算向量）。
_fetch_candidate_embeddings取候选向量：只读缓存，绝不在热路径调接口。
_select_with_mmr / MMR多样性选取：在「相关度」和「与已选的相似度」之间权衡。
batch_insert_recommendations批量写推荐历史（唯一的同步写，要拿 ID 返回）。
mark shown把选中项标记为「已展示」，脱离式后台执行、不占响应时间。
③ 评分与多样性
PoolCurator候选池二次评分器：综合五维算 rec_score 覆盖排序。
rec_score综合推荐分 = 相关 0.30 + 新鲜 0.20 − 话题疲劳 0.25 − 来源单调 0.15 + 惊喜 0.20，再加反馈调整。
topic_fatigue / source_monotony / serendipity话题疲劳（同话题看多了降权）/ 来源单调（同来源太多降权）/ 惊喜度（跨域加分）。
topic / style / broad cap每话题 / 每风格 / 每广义话题在一批里的数量上限，保证多样性。
amplification guard 放大保护新确认的兴趣每批限量（floor(limit×0.25)），防止一个方向刷屏；当前 serve() 里默认休眠。
topic supergroup 话题超组把近义话题归并成一桶后再做多样性控制。
④ 后台预计算
classify_pool_backlog源无关补分类：给跨源、没经发现引擎的内容补话题 / 风格 / 相关度。
precompute_pool_copy预生成推荐文案 / 话题标签；限流则留空下轮重试。
precompute_delight_scores / LLMDelightScorer惊喜打分：大模型按「跨域惊喜 / 隐藏需求」给候选打分。
delight主动「惊喜」推荐：跨域、可能戳中隐藏需求的内容（非常规相关推荐）。
warm / prewarm embeddings预热向量缓存，让 serve() 的 MMR 能直接命中缓存。
⑤ 反馈与回流
recommendation-click推荐点击事件入口（优先用回传的真实 URL + 平台）。
like / dislike / comment / dismiss喜欢 / 不喜欢 / 评论 / 忽略。
dismiss 忽略软移除：不更新画像、不升级成话题级负反馈。
get_pending_delight取一条可推送的惊喜候选（分≥0.70、4 小时冷却、过滤已讨厌话题）。
process_feedback_batch_if_needed（Soul 侧）攒够反馈后重跑偏好分析 —— 推荐反馈最终回流到画像。
