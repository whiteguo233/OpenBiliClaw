1. 架构主图：输入、编排、存储、消费者

输入来源

XHS / 抖音 / YouTube bootstrap_profile

插件回传账号信号后转成统一 event，再在初始化完成后推入 ProfileUpdatePipeline。

  • XHS: saved / liked / history
  • Douyin: post / collect / like / follow
  • YouTube: history / subscriptions / likes

推荐点击

/api/recommendation-click 同时落 click event，并额外构造 RECOMMENDATION_CLICK 强画像信号。

后台账户同步

AccountSyncService 默认每 6 小时增量同步 B 站 history / favorites / following；有新增事件时直接调用 analyze_events()。

反馈与对话旁路

/api/feedback 和 learn_from_dialogue() 当前不走 pipeline，而是各自触发偏好重析或候选合并。

手动编辑 / 探针响应

画像编辑写覆盖层；兴趣/避雷探针先写 speculator 状态，确认达标后通过 tick 或 force_tick 转正。

初始化 / 手动 rebuild

显式消费历史事件和 preference，直接运行 analyze_events() 与 build_initial_profile()。

SoulEngine 编排核心

ProfileUpdatePipeline

classify_signal() 将信号分到 Surface / Interest / Role / Values / Core，按每层 buffer 与阈值触发更新。

  • Surface: 3 条 / 5 分钟
  • Interest: 3 条 / 10 分钟
  • Role: 5 条 / 24 小时
  • Values: 5 条 / 24 小时
  • Core: 8 条 / 48 小时

Runtime tick

ContinuousRefreshController.run_forever() 启动 _loop_soul_pipeline()，默认每 60 秒调用 pipeline.tick()；受 scheduler gate、插件断开暂停和 init-active gate 控制。

按层 Updater

Surface 用规则；Interest 调 PreferenceAnalyzer；Role / Values / Core 调 LLM delta prompt；Core 或 Values 变化后重生成 portrait。

CognitionCycle

12 小时节流生成 awareness 与 insight，并只把最近窗口同步进 Soul，不主动完整 rebuild。

Speculator / Consolidator

兴趣探针转正、避雷转正、12 小时画像整理都可能局部写回 Soul 的兴趣树和避雷树；配置热重载后还会 one-shot force_tick 生成/轮换探针。

持久化层

event list

事实记录。普通 /api/events 只落这里，不自动喂 pipeline。

preference.json

flat preference：兴趣、避雷、风格、上下文、source mix。Interest updater 和反馈/对话重析都会写它。

soul.json

五层 OnionProfile：只有 updater 返回 changed=true、完整 rebuild、或认知窗口同步时才写。

pipeline_state.json

保存每层 buffer、上次更新时间和 update_count，防止重启丢掉未处理信号。

runtime state

保存 probe feedback history、账户同步游标、cognition cycle 游标、consolidation run 记录等后台状态。

profile_overrides.json

手动编辑覆盖层。它不改 raw Soul，但 get_profile() 读时会叠加，业务上优先级最高。

下游消费者

Recommendation

读取有效画像的兴趣、避雷、source mix、active speculations，影响候选筛选和排序。

Discovery

使用兴趣、避雷、探针状态来生成搜索词、清池、补候选池。

画像页 / OpenClaw

展示 soul_profile.md、cognition cards、changelog 和编辑覆盖层。

2. Soul 写回路径地图

FULL

完整 rebuild

  1. 初始化或手动 rebuild：build_initial_profile()。
  2. Dialogue insight 达学习阈值 -> PreferenceAnalyzer -> preference 显著变化 -> ProfileBuilder。
  3. Feedback batch 达阈值 -> PreferenceAnalyzer -> preference 显著变化 -> ProfileBuilder。
LOCAL

局部写回

  1. ProfileUpdatePipeline 更新 Surface / Interest / Role / Values / Core。
  2. Core 或 Values 层变化后，只重生成 personality_portrait。
  3. Speculator 转正、avoidance 转正、consolidation 整理兴趣树。
LOOP

后台定时触发

  1. _loop_soul_pipeline() 默认 60 秒 tick 一次，flush buffer 并跑低频维护。
  2. AccountSyncService 默认 6 小时同步账号信号，更新 preference；画像为空时自动 bootstrap。
  3. CognitionCycle 与 ProfileConsolidator 默认 12 小时节流；speculator 空闲默认 30 分钟检查生命周期。
SYNC

同步与覆盖

  1. CognitionCycle 同步最新 8 条 awareness 与 6 条 insight 到 Soul。
  2. Insight feedback 只同步 active insight 的验证状态。
  3. ProfileOverrides 不改 raw Soul，但改变 effective profile。

3. 名词注释

ProfileSignal

进入 pipeline 的标准信号，包含 signal_type、payload、target_layers 和置信度。

LayerBuffer

每个 Onion layer 独立的信号缓冲。未达阈值的信号留在这里，等待下一次 ingest 或 tick。

PreferenceAnalyzer

LLM 结构化任务，把事件批次转换成 flat preference；Interest updater 再将其填回 OnionProfile。

OnionProfile

五层画像：Surface、Interest、Role、Values、Core，加上 portrait、awareness、insight 窗口。

changed=true

LayerUpdater 的 diff 判定。只有有实质变化时，pipeline 才保存 Soul 并追加 changelog。

effective profile

业务读取的最终画像：raw Soul + 用户手动 overrides + active speculations。

runtime loop

后端启动后的 asyncio 定时循环；当前 Soul 调度不依赖 discovery_cron。

实现边界：当前普通 /api/events、Bili source task-result、X fetch、YouTube Takeout import 都不会自动进入 ProfileUpdatePipeline。它们要么只落 memory，要么走初始化/手动重析旁路。账户同步是例外：它不进 pipeline，但会在后台定时直接调用 analyze_events() 更新 preference。
