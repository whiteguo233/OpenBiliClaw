下面把四个来源「从触发到入池」的完整流程拆成步骤讲清楚。英文术语首次出现时用 等宽蓝字 标出并就地加中文解释；全部术语的对照见 §9。一句话总览：哔哩哔哩、抖音(直采)、YouTube 都走"发现引擎"统一抓取打分；小红书、抖音(扩展) 则在你自己的浏览器里采集、绕过引擎，再由推荐侧用同一套评估 prompt 补打分（见 §7）。除哔哩哔哩外其余三个来源默认关闭，需在配置里开启。

▸ 哔哩哔哩 (Bilibili) 默认开启 主来源 · 经发现引擎
  1. 什么时候触发？ 后台有个每 60 秒跑一次的常驻循环 _loop_refresh（刷新循环）。它先看候选池够不够： ① 可用量 < 300（pool_target_count，候选池目标库存量）→ 进入「补货模式」，按各平台配额补；
    ② 池子已满时，只要满足三个独立条件之一就抓：累计新行为事件 ≥ 6（看 / 搜 / 收藏 / 点赞 等）、距上次热门刷新 > 3 小时、距上次跨域探索 > 12 小时。
  2. 怎么抓？ 调用 ContentDiscoveryEngine.discover()（内容发现引擎的主入口），把多个策略并发跑（asyncio.gather，同时执行多个任务）。默认是「两段式」：先单独跑 search，停 2 秒避开 B 站风控，再并发跑其余三个。所有策略都用真实的 BilibiliAPIClient（B 站官方接口客户端）抓数据。
  3. 四个策略（每个都先抓候选、再用大语言模型 LLM 逐条打相关度分）： • search 搜索：大模型按画像生成搜索词 → 搜索 → 打分，保留 ≥ 0.60；
    • trending 热门：大模型挑分区 → 拉排行榜 → 打分 ≥ 0.60；
    • related_chain 相关链：拿你看过 / 收藏的视频当「种子」，沿「相关推荐」做 BFS（广度优先遍历）最多 2 层 → 打分 ≥ 0.60；
    • explore 探索：大模型生成你舒适区之外的新领域 → 搜索 → 打分（带新颖度加成），阈值 0.58。
  4. 汇总成池：_merge_and_rank（去重 + 排序）→ _normalize_topic_*（用向量把同义话题归并）→ _compress_topic_repeats（同话题 / 风格限量，保证多样性）→ 数量不够则 _run_backfill（放宽阈值重抓 / 取历史缓存）→ _cache_results 写入候选池 content_cache。
  5. 结果：这些视频在引擎里已经打好分、分好话题 / 风格，可直接进推荐，无需后续补分类。
▸ 小红书 (Xiaohongshu) 默认关闭 纯扩展采集 · 绕过发现引擎
  1. 为什么不一样？ 小红书风控严，后端绝不直接调它的接口、也不爬取。所有抓取都在你自己的浏览器里、用你的真实登录态完成。后端的 XiaohongshuAdapter（小红书适配器）是个空壳占位（no-op stub，调用直接返回空），只为让系统「认得」这个来源、不必到处写特例。
  2. 第一步 · 后端出题：XhsTaskProducer（小红书任务生产者）每 1 小时跑一次，读你的画像，让大模型把兴趣改写成小红书风格的搜索词（generate_xhs_keywords）。它只把搜索词当作「任务」入队，存进 XhsTaskQueue（任务队列，一张 SQLite 表，任务状态：待处理 → 进行中 → 完成 / 失败）。
  3. 第二步 · 浏览器答题：Chrome 扩展轮询 GET /api/sources/xhs/next-task（领一个任务）→ 在后台标签页打开小红书搜索结果 → 直接从网页元素 DOM 里读卡片的标题 / 作者 / 封面（extractNoteMetadataFromAnchor，零额外网络请求）→ 通过 POST .../task-result 和 .../observed-urls 回传。⚠ 严禁自动滚动 —— 自动滚动是小红书识别机器人的典型信号，扩展只读你自己已经滚到的卡片。
  4. 第三步 · 写入池：后端 _cache_xhs_notes() 把回传的笔记直接写进 content_cache，不经过发现引擎（所以没有引擎里那套打分 / 多样性处理）。
  5. 第四步 · 补分类：因为没经引擎，推荐侧用 classify_pool_backlog()（源无关补分类）给这些笔记补上话题 / 风格 / 相关度分。注意：它用的是和发现侧 evaluate_content_batch 完全相同的评估 prompt（build_batch_content_evaluation_prompt），只是调用方变成 RecommendationEngine（caller 标签 recommendation.evaluate_batch）—— 同一把尺子、不同工位，并不是另一套更弱的评估。补完之后就和 B 站内容一视同仁地进推荐。
  6. 另外两条补充通道：① 被动收集 —— 你自己刷小红书时，扩展顺手采集视口里出现的卡片；② 创作者订阅 —— 每 24 小时去你关注的创作者主页抓最新笔记。
▸ 抖音 (Douyin) 默认关闭 双通道：后端直采 + 扩展采集

抖音有两条入池通道，按配置走其一：

  1. 通道 A · 后端直采（需登录凭据 cookie 且配置 mode=direct）：DouyinDiscoveryProducer（抖音生产者，每 30 分钟）→ 用 DouyinDirectClient（抖音直连客户端，带你的 cookie）跑 DouyinDirectStrategy（覆盖搜索 / 热榜 / 推荐流 / 创作者）→ 临时 register_strategy（把该策略注册进引擎）后调 engine.discover([douyin_direct])。这条和哔哩哔哩一样经过发现引擎，自带打分、去重、多样性和入池。
  2. 通道 B · 扩展采集（可选，和小红书同款思路）：任务进 DyTaskQueue（抖音任务队列），扩展在后台标签页用 DOM 提取后回传 → 直接写 content_cache，绕过发现引擎，同样靠 classify_pool_backlog 补分类。
▸ YouTube 默认关闭 后端直采 · 经发现引擎
  1. 触发：YoutubeDiscoveryProducer（YouTube 生产者）每 60 分钟跑一次，按平台配额缺口决定要不要抓。
  2. 怎么抓：用 YtScraperClient（YouTube 抓取客户端，底层是开源库 scrapetube / yt-dlp，不需要官方 API key）跑三个策略： • yt_search 搜索（大模型本地生成查询词）；
    • yt_trending 趋势榜；
    • yt_channel 订阅频道（从你的「关注」事件里读频道 ID）。
  3. 入池：逐个策略 register_strategy + engine.discover([yt_*])，和哔哩哔哩一样经过发现引擎打分入池。
  4. 说明：稳态运行是后端直采；只有首次初始化（bootstrap，导入历史时）才借用扩展。
9. 术语对照表 — 本页英文速查

下表解释本页（以及 soul / recommendation 两张图）里出现的主要英文标识，按用途分组。看图时遇到看不懂的英文，对照这里即可。

英文标识中文含义 / 作用
① 发现引擎与调度
ContentDiscoveryEngine.discover()内容发现引擎的主入口：把各策略跑一遍，再去重、归一、压缩、补货、写入候选池。
ContinuousRefreshController持续刷新调度器：常驻后台，决定什么时候发现 / 补货 / 清理。
run_forever / _loop_*常驻后台循环。run_forever 同时启动 7 个 _loop_xxx（刷新、文案预算、画像 tick、三个来源生产者、主动推送）。
refresh_if_needed每一轮判断「现在到底要不要刷新」的总闸门。
asyncio.gatherPython 并发原语：同时跑多个异步任务、一起等结果。
fan-out并行分发：多个策略一起跑（而不是排队一个个跑）。
producer 生产者各来源独立的「入池调度器」，如 XhsTaskProducer / DouyinDiscoveryProducer。
content_cache候选池：一张 SQLite 表，存放所有等待推荐的内容。
pool_target_count候选池目标可用量（默认 300）；低于它就触发补货。
pool_source_shares各来源配额比例 B站:小红书:抖音:YouTube = 8:1:1:1；关掉的源配额会被剔除并重新分配。
discovery_limit每个策略单次抓取上限（默认 30，最多 60）。
backfill / _run_backfill补货：数量不够时放宽分数阈值重抓，或从历史缓存里取。
DiscoveredContent「发现到的一条内容」的数据对象（标题 / 分数 / 话题 / 风格 / 来源等字段）。
② 策略与打分
BilibiliAPIClientB 站官方接口客户端 —— 生产环境真实抓取用的就是它。
MockBilibiliClient模拟客户端，仅离线评估 (§3) 使用，生产环境从不使用。
LLM大语言模型：这里用来生成搜索词、给内容打相关度分、写推荐文案等。
evaluate_content_batch批量内容打分：把候选成批交给大模型逐条评相关度（发现侧，caller=discovery.evaluate_batch）。同一套 prompt 也被推荐侧 classify_pool_backlog 复用，给扩展直采来源（小红书 / 抖音扩展）补分。
score_threshold保留阈值，低于此分的内容被丢弃。普通 admission floor = 0.60；策略可更严格，explore = 0.58（backfill 最低 0.55）。
_merge_and_rank合并去重 + 排序（按层级、分数、播放量）。
_normalize_topic_groups / _keys话题归一化：用向量把「动漫 / 动漫产业 / 动漫文化」这类同义话题合并成一个。
_compress_topic_repeats多样性压缩：同一话题 / 风格 / 来源限量，避免一批内容太同质。
_cache_results把最终结果写入候选池 content_cache。
last_intermediates策略的「中间产物」：生成的搜索词 / 分区 rid / 种子 / 探索领域，便于排查。
BFS广度优先遍历：related_chain 沿「相关视频」一层层向外扩展时用。
embedding / MMR向量 / 最大边际相关：推荐侧用向量相似度做多样性选取。
③ 各来源专有（小红书 / 抖音 / YouTube）
XhsTaskQueue / DyTaskQueue / YtTaskQueue小红书 / 抖音 / YouTube 各自的任务队列（独立 SQLite 表，状态：待处理→进行中→完成/失败）。
XhsTaskProducer小红书任务生产者：把画像兴趣改写成搜索词、入队（自己不调接口）。
generate_xhs_keywords让大模型把你的兴趣改写成小红书风格的搜索关键词。
next-task / task-result / observed-urls扩展用的接口：领任务 / 回传任务结果 / 上报浏览中观察到的链接。
_cache_xhs_notes后端把扩展回传的小红书笔记直接写入候选池（绕过引擎）。
classify_pool_backlog源无关补分类：给扩展直采、没经引擎的内容补话题 / 风格 / 相关度。用的是和发现侧 evaluate_content_batch 完全相同的评估 prompt，只是 caller 改为 recommendation.evaluate_batch（同一把尺子、不同工位）。
XiaohongshuAdapter (no-op stub)小红书适配器占位，空实现（fetch 直接返回空）；只为让系统「认得」这个来源。
AdapterRegistry / SourceAdapter来源适配器注册表 / 统一协议；目前实际注册的只有 bilibili 与 xiaohongshu(占位)。
DouyinDirectClient / DouyinDirectStrategy抖音直连客户端 / 直采策略（带 cookie，覆盖搜索/热榜/推荐流/创作者）。
YtScraperClient (scrapetube / yt-dlp)YouTube 抓取客户端，基于开源库，无需官方 API key。
register_strategy把某个策略临时注册进引擎，再单独跑它（抖音/YouTube 走这条）。
DOM / extractNoteMetadataFromAnchor网页元素 / 从卡片元素里提取标题、作者、封面（零额外请求）。
cookie / mode=direct登录凭据 / 抖音「后端直采」模式开关。
bootstrap首次初始化：导入历史数据、生成初始画像的一次性流程。
④ 跨图通用（recommendation / soul）
RecommendationEngine.serve()推荐主入口；换一批(reshuffle) / 续页(append) / 生成(generate) 都委托给它。
PoolCurator候选池二次评分器：综合新鲜度 / 话题疲劳 / 来源单调 / 惊喜度 / 反馈再排序。
SoulProfile / OnionProfile用户画像；OnionProfile 是当前的五层「洋葱模型」（核心/价值/兴趣/角色/表层）。
speculator / speculation猜测兴趣探针：主动猜你可能感兴趣的新方向，命中后才写进画像。
