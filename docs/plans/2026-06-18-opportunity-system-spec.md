# 机会系统统一规格：画像准确性 + OpenCloud / HMA / WorkValue 客户端

**Created:** 2026-06-18  
**Status:** Draft for review  
**Source context:** `docs/spec.md`, `docs/architecture.md`, `docs/openclaw-quickstart.md`,
`docs/modules/soul.md`, `docs/modules/memory.md`, `docs/modules/integrations.md`,
profile taxonomy / editable-profile / interest-probe specs

## 1. 目标

把 OpenBiliClaw 现有的“本地跨平台内容发现 Agent”整理成一个更完整的机会系统：

1. **先把画像做准**：从行为、对话、显式编辑、探针确认和长期反馈中生成可解释、可校准、可回滚的用户画像。
2. **再把画像变成外部系统可用能力**：通过 OpenCloud / OpenClaw、HMA 和 WorkValue 客户端，把画像、追问、推荐和反馈闭环输出成稳定接口。
3. **最后用一个 HTML 讲清楚我们要做什么**：面向合作者、投资人或内部团队，用一页可视化叙事讲清系统定位、架构、优先级和落地路线。

本文先锁定“做什么”和“怎么组织系统”，不进入具体代码实施。

## 2. 术语和当前假设

仓库当前已有 `OpenClaw` 集成文档和实现，但没有 `OpenCloud`、`HMA`、`WorkValue`
的定义。因此本 spec 先采用以下边界：

| 术语 | 本 spec 中的定义 | 状态 |
|------|------------------|------|
| OpenBiliClaw Core | 本地画像、发现、推荐、学习和运行时主链 | 仓库已实现 |
| OpenCloud | 可选的云端/协同层：分发、设备同步、模型路由、团队集成、远程调用治理；不默认持有裸画像 | 本文临时边界 |
| OpenClaw | AI Agent / 编码助手入口，通过 workspace skill + JSON CLI bridge 调本地能力 | 仓库已实现 |
| HMA | Human Model Agent/API：面向外部系统消费“人的模型”的抽象层，读取画像摘要、提出澄清问题、写回反馈 | 本文临时边界 |
| WorkValue 客户端 | 用户侧工作价值客户端：把个人画像用于工作流、内容、任务和价值判断场景 | 本文临时边界 |

如果用户口中的 **OpenCloud 实际指 OpenClaw**，本文中的“OpenCloud 协同层”可以收窄为
“OpenClaw 集成层 + 本地 API + workspace skill”，系统结构不变。

## 3. 机会判断

OpenBiliClaw 现在不只是“推荐工具”，而是一个可以沉淀长期个人理解的本地 Agent
内核。机会不在于再做一个内容流，而在于把“对一个人的理解”做成可持续更新、可解释、
可调用的个人模型。

核心机会分三层：

1. **个人画像资产**：用户长期行为、对话和反馈不是一次性 prompt 上下文，而是逐层沉淀成可审计的画像。
2. **主动校准机制**：系统不只被动猜，而是通过兴趣探针、避雷探针、苏格拉底式对话和手动编辑主动逼近真实偏好。
3. **外部工作流接入**：画像一旦稳定，就能被 OpenClaw / HMA / WorkValue 客户端调用，用于推荐、任务选择、内容判断、学习路径、职业/工作价值建议等场景。

系统边界原则：

1. **画像是核心资产**：所有外部入口都围绕画像读写，而不是各自发明一套偏好系统。
2. **OpenBiliClaw Core 仍是事实源**：外部系统不直接改 SQLite / JSON 内部文件，只通过稳定 API / adapter 操作。
3. **OpenCloud 不默认拿走裸数据**：除非用户明确启用同步，否则云端只做任务、设备、分发、模型路由或加密同步协调。
4. **HMA 只消费裁剪后的人的模型**：对外提供摘要、证据、置信度和追问，不暴露内部事件明细。
5. **WorkValue 客户端负责场景化表达**：它不是第二套画像引擎，而是把画像用于工作价值、内容选择和行动建议。

## 5. 第一优先级：画像怎么做准

画像准确性是 P0。没有准画像，后面的 OpenCloud、HMA、WorkValue 都只是在放大噪声。

### 5.1 当前可复用能力

已有基础包括：

- 五层记忆：Event → Preference → Awareness → Insight → Soul。
- 固定一级分类词表：减少 LLM 自由分类漂移。
- 同名异义防护：避免“苹果(科技)”和“苹果(食物)”误合并。
- 用户覆盖层：手动编辑不会被画像重建覆盖。
- 正向兴趣探针：主动确认潜在兴趣，并区分 near / lateral / bridge / wildcard。
- 避雷探针：未确认前不进入长期过滤，确认后写入 `disliked_topics` 并清池。
- 短期探索 buffer：弱正向先试探，不立刻升成长期兴趣。
- 对话学习：聊天进入受控学习链路。

### 5.2 画像准确性的产品定义

“准”不是只有推荐点击率，而是四个维度同时成立：

| 维度 | 定义 | 失败表现 |
|------|------|----------|
| 事实准 | 行为、反馈、来源、时间和上下文记录正确 | 把误点、短停留、旧兴趣当强信号 |
| 分类准 | 兴趣、避雷、价值观、风格放在正确层级 | 大量同义分类、同名异义误合并 |
| 解释准 | 能说清为什么这么理解用户 | 画像像标签堆砌，无法追溯证据 |
| 更新准 | 新反馈能改变该改变的部分，不造成过拟合 | 一次“多来点”导致推荐塌缩 |

### 5.3 P0 画像准确性路线

1. **建立画像质量评估面板**
   - 展示 raw vs effective profile。
   - 标出每个画像结论的来源：行为、聊天、探针、手动编辑、整理器。
   - 记录置信度、最近证据、最后更新和被用户修正次数。

2. **建立“证据到结论”的可追溯链**
   - 每个重要画像项保留 top evidence。
   - 推荐理由引用画像结论时能反查证据摘要。
   - 用户否定画像项时，把否定写入对应证据链，而不是只删文案。

3. **把准确性闭环产品化**
   - 画像页提供“准 / 不准 / 需要更细”反馈。
   - 不准反馈触发小范围澄清问题，而不是整张画像重建。
   - 画像编辑后同步影响探针、发现、推荐、避雷和 OpenClaw/HMA 输出。

4. **区分长期画像和短期意图**
   - 长期画像：稳定偏好、价值观、认知风格、避雷。
   - 短期意图：今天想看什么、最近在忙什么、当下工作任务。
   - WorkValue 客户端默认先读长期画像，再叠加当前会话意图。

5. **定义画像质量指标**
   - 用户主动修正率：越低越好，但早期应鼓励暴露问题。
   - 修正后复发率：用户删掉的项是否又被 AI 推回来。
   - 探针确认率：正向 / 避雷探针是否真的问到有价值方向。
   - 推荐解释采纳率：用户是否认可“为什么推这个”的解释。
   - 过拟合报警：单一 topic/source 在推荐池占比异常升高。

## 6. 第二优先级：OpenCloud / HMA / WorkValue 客户端一起工作

P1 目标是把“准画像”变成外部系统可用的稳定能力。

### 6.1 集成结构

```text
WorkValue Client
  ├─ 读：profile summary / work context / recommendations
  ├─ 写：explicit feedback / edits / current task intent
  └─ 问：clarification question / HMA dialogue

HMA Interface
  ├─ profile_read: 裁剪后的用户模型
  ├─ profile_explain: 画像结论 + 证据摘要
  ├─ clarify: 生成一个高价值澄清问题
  └─ feedback_write: 把用户回答回写到学习链路

OpenCloud / OpenClaw
  ├─ agent skill discovery
  ├─ JSON CLI bridge / API bridge
  ├─ WebSocket proactive events
  ├─ optional device sync / deployment
  └─ model routing / auth / usage governance

OpenBiliClaw Core
  └─ 唯一画像、发现、推荐和反馈事实源
```

### 6.2 P1 接口契约

第一版只需要四类能力：

| 能力 | 说明 | 当前基础 |
|------|------|----------|
| `get-profile` | 读取裁剪画像、兴趣、避雷、风格、最近认知变化 | OpenClaw CLI 已有 |
| `explain-profile-item` | 解释某个画像项为什么成立，返回证据摘要与置信度 | 待新增 |
| `ask-clarifying-question` | 基于低置信/高价值画像缺口生成一个问题 | 可复用 SocraticDialogue / probes |
| `write-feedback` | 写回回答、编辑、喜欢、不喜欢、工作上下文 | OpenClaw feedback/chat 已有部分 |

P1 不直接开放内部数据库。外部系统只拿 DTO：

```json
{
  "user_model": {
    "stable_traits": [],
    "interests": [],
    "avoidances": [],
    "values": [],
    "work_context": [],
    "confidence": {}
  },
  "evidence": {
    "summary": "为什么这样判断",
    "sources": ["event", "dialogue", "probe", "manual_edit"]
  },
  "next_best_question": {
    "question": "最值得问用户的一句话",
    "reason": "这个问题能提升哪个画像维度"
  }
}
```

### 6.3 WorkValue 客户端职责

WorkValue 端不应复制推荐系统，而是做三个场景化入口：

1. **工作价值画像页**
   - 用户现在偏好的任务类型、信息密度、协作方式、风险偏好。
   - 哪些判断来自长期行为，哪些来自最近对话。

2. **工作流建议**
   - “今天/这周我应该看什么、做什么、避开什么”。
   - 把内容推荐升级成行动建议，但每条建议必须有画像依据。

3. **校准入口**
   - 用户可以直接纠正画像。
   - 客户端主动展示高价值澄清问题。
   - 回答回写 HMA/OpenBiliClaw，而不是只留在前端。

## 7. 后续优先级建议

| 优先级 | 主题 | 原因 | 产物 |
|--------|------|------|------|
| P0 | 画像准确性与可解释校准 | 根基；不准会污染所有外部系统 | 画像质量面板、证据链、校准 API |
| P1 | OpenCloud / HMA / WorkValue 集成契约 | 把画像变成可调用能力 | DTO、桥接命令、客户端交互流 |
| P2 | HTML 叙事页 | 对外讲清楚“我们要做什么” | 单页 HTML + 架构图 + 路线图 |
| P3 | 多源信号质量治理 | 提高画像输入质量 | 来源可靠性、信号强弱、去重、冷却 |
| P4 | 推荐与工作价值场景闭环 | 把准画像转化成结果 | 推荐、任务建议、用户采纳反馈 |
| P5 | 云端/多端治理 | 扩展规模与部署便利性 | 同步、安全、权限、成本与模型路由 |

## 8. HTML 叙事页规格

HTML 不是文档搬运，而是一页“我们在建什么”的可视化说明。建议结构：

1. **首屏：一句话定位**
   - “把长期个人理解做成可校准、可调用的本地 Agent 内核。”
   - 第一屏同时出现系统图，不做纯 marketing hero。

2. **问题：现在的推荐和 Agent 缺什么**
   - 推荐懂点击，不懂人。
   - Agent 有上下文，没有长期可验证画像。
   - 工作流工具缺少“这个人真正适合什么”的底层模型。

3. **解决方案：五层画像 + 主动校准 + 外部调用**
   - Event → Preference → Awareness → Insight → Soul。
   - Probes / Dialogue / Manual edits。
   - OpenCloud / HMA / WorkValue 调用画像。

4. **系统结构图**
   - 左侧：信号来源和客户端。
   - 中间：OpenBiliClaw Core。
   - 右侧：OpenCloud / HMA / WorkValue。
   - 底部：反馈闭环。

5. **优先级路线**
   - P0 画像做准。
   - P1 集成契约。
   - P2 HTML 和 Demo。
   - P3+ 扩展。

6. **成功标准**
   - 用户能看懂画像并纠正。
   - 系统能解释为什么这么理解。
   - WorkValue 能基于画像给出有用建议。
   - OpenClaw/HMA 能稳定读写，不绕过主链。

视觉风格建议：工作型、克制、信息密度高；不要做大面积装饰渐变和营销卡片。重点是系统图、
流程、优先级和证据链，让人快速理解这是“个人模型基础设施”，不是普通推荐 App。

## 9. 开放问题

这些问题不阻塞当前 spec，但需要在进入实现计划前锁定：

1. **OpenCloud 的真实边界**：是 OpenClaw 的口误，还是一个独立云端产品？
2. **HMA 的全称与拥有方**：它是内部 API、外部系统，还是一个独立 Agent？
3. **WorkValue 客户端首个场景**：内容推荐、任务选择、职业/工作价值分析，还是团队协作？
4. **数据策略**：OpenCloud 是否允许保存加密画像摘要，还是只做本地代理和任务分发？
5. **HTML 受众**：给用户、开发者、投资人还是内部团队？不同受众会影响叙事重点。
